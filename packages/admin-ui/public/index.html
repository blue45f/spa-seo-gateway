라우트 오버라이드 — URL 패턴별 동작 분기

URL 의 pathname + search 에 정규식이 매칭되면 해당 라우트의 설정이 글로벌 설정을 오버라이드합니다. 위에서 아래로 첫 매칭이 승리하므로 더 구체적인 패턴을 위로 두세요.

필드 가이드
  • pattern: 정규식. 예) ^/$ (홈), ^/products/ (제품 하위), ^/(account|cart)(/|$) (인증 페이지)
  • ttl: 라우트별 캐시 수명 (ms). 빠르게 변하는 페이지(홈, 검색결과)는 짧게(60_000–600_000), 정적 페이지(블로그)는 길게(86_400_000)
  • waitUntil: networkidle2 가 가장 안전(기본). 데이터 늦게 오는 SPA 는 networkidle0, 빠른 정적 페이지는 domcontentloaded
  • selector: 특정 DOM 요소가 나타날 때까지 대기. 예) [data-product-loaded]
  • waitMs: waitUntil 충족 후 추가 대기. 애니메이션/지연 데이터 로딩 시
  • ignore: true 면 렌더하지 않고 204 반환. 인증 필요 페이지/내부 라우트에
예시 
^/(account|admin|cart|checkout)(/|$)  ignore=true             ← 인증/결제는 렌더 금지
^/products/[0-9]+                      ttl=21600000  selector=[data-product]  ← 6h
^/blog/                                ttl=86400000             ← 24h
^/                                     ttl=600000               ← 10분 (홈)

💾 저장 (메모리)는 즉시 적용되지만 재시작 시 사라집니다. 저장 + 디스크 영구화는 seo-gateway.config.json 에 기록.

Route overrides

pattern 은 URL 의 pathname + search 에 적용되는 정규식입니다. 빈 라우트는 자동 무시됩니다.

캐시 — 렌더 결과의 보관과 무효화

기본 24시간 TTL 안에서는 같은 URL 의 봇 요청은 캐시에서 즉시 응답. TTL 만료 후 1시간 SWR 윈도우 동안엔 stale 응답을 즉시 주고 백그라운드에서 갱신.

언제 무효화?
  • 배포 직후 — 빌드 파이프라인에 POST /admin/cache/clear 추가 권장
  • 특정 페이지 콘텐츠 수정 — URL 입력 + URL 무효화
  • Search Console 에서 인덱스 누락 발견 시 — 해당 URL 만 무효화하고 봇 재방문 유도

⚠️ 전체 초기화는 다음 봇 트래픽 시 모든 페이지가 cold render 됩니다. 트래픽 큰 사이트에선 신중히.

Sitemap 사전 워밍 — cold path 제거

              

                sitemap.xml 을 파싱해 안에 적힌 URL 들을 미리 렌더링합니다. 결과는 캐시에 저장되어
                실제 봇 트래픽이 들어왔을 때 즉시 hit. cold render 처음 한 번의 지연을 미리 흡수.
              

              

                언제 실행?
                
    
                  
  • 배포 직후 — CI 파이프라인에서 cache/clear 다음 단계로 호출
    • 
                  
  • 주기적 — cron 으로 매 N시간마다 실행 (TTL 만료 전에 미리 갱신)
    • 
                  
  • 대규모 콘텐츠 추가 후 — 신규 URL 들이 즉시 봇에게 노출되도록
    • 
                

              

              

                파라미터
                
    
                  
  • max: 워밍할 최대 URL 수 (sitemap-index 의 모든 sub 합산). 기본 1000
    • 
                  
  • concurrency: 동시 렌더 수. POOL_MAX 보다 작아야 일반 봇 트래픽도 받음. 기본 4
    • 
                

              

              

                ℹ️ sitemap-index (여러 sitemap 묶음) 도 자동 재귀 파싱. 큰 sitemap 은 시간이 걸리므로 백그라운드 실행.

렌더 테스트 — 디버깅 도구

              

                URL 을 입력하면 풀에서 페이지를 잡아 즉시 렌더링하고, 결과 HTML 의 첫 4KB 를 미리보기.
                캐시 우회 — 새 렌더가 항상 실행됩니다.
              

              

                활용 시나리오
                
    
                  
  • 새 라우트 추가 후 — waitUntil/selector 가 의도대로 동작하는지 확인
    • 
                  
  • 모바일 vs 데스크톱 차이 검증 — UA 칸에 모바일 UA 넣어 viewport 차이 확인
    • 
                  
  • quality gate 트리거 확인 — soft 404 페이지로 테스트해 status 가 자동 다운그레이드 되는지
    • 
                  
  • 응답 헤더 확인 — x-prerender-route, x-prerender-viewport, x-prerender-quality
    •
